# PadMapper

PadMapper 是一个面向 macOS 的本地小键盘映射工具原型。

它的目标是把一块独立小键盘当作“功能板 / 宏键盘”来使用：识别目标设备、读取真实按键、把若干键固定成一个按键组，再把这个按键组映射到快捷键或多媒体键动作。

当前版本已经打通了这条主链路：

- 发现并选择真实 HID 键盘类设备
- 读取真实按键输入
- 把多个物理键固定成一个按键组
- 给按键组配置快捷键或多媒体键
- 命中后向系统发送动作
- 最小化到托盘，在后台继续运行

## 当前产品语义

当前版本里的“按键组”不是传统意义上的“必须同时按下的组合键”。

实际语义是：

- 你可以把多个物理键归为同一个按键组
- 这个组里的任意一个键按下，都可以触发这组动作
- 如果组内多个键几乎同时或短时间内先后按下，只会整体触发一次
- 一个物理键只能属于一个按键组

这套语义是当前实现的核心，请不要按“多键必须同时按下”的思路理解它。

## 当前功能

### 已实现

- macOS 14+ 原生 SwiftUI 桌面应用
- 真实 HID 键盘/小键盘发现与选择
- 监听单一目标设备输入
- 物理键码实时识别
- 按键组创建、删除、重命名
- 快捷键动作
- 多媒体键动作
- 冲突检测
  - 如果某个物理键已经属于另一个按键组，会阻止保存并提示冲突
- 右侧运行反馈面板
- 配置持久化到本地
- 菜单栏托盘常驻

### 当前限制

- 目前是“监听模式”，不是“独占设备模式”
  - 也就是说，目标小键盘原始输入不一定会被完全拦截
- 当前只支持单配置、单设备主流程
- 当前动作类型仅支持：
  - 快捷键
  - 多媒体键
- 不支持：
  - 打开 App
  - Shell 命令
  - 快捷指令
  - 多 Profile
  - 按 App 切换规则
  - 导入导出
- UI 已经过一轮产品化整理，但仍然属于原型阶段

## 使用方法

### 基本流程

1. 打开应用，确认顶部显示了真实设备或测试模式状态
2. 在目标小键盘上按下你想归为一组的键
3. 点击“固定当前按键组”
4. 在右侧填写按键组名称
5. 选择动作类型
6. 配置快捷键或多媒体键
7. 点击“确认配置”
8. 再按该组中的任意一个键，测试动作是否生效

### 配置快捷键

例如配置“复制”：

1. 在右侧把按键组名称改成 `copy`
2. 动作类型选择“快捷键”
3. 按键填写 `c`
4. 勾选 `Cmd`
5. 点击“确认配置”

这时该按键组中的任意一个键，都会触发 `Cmd+C`。

### 配置多媒体键

例如配置“播放 / 暂停”：

1. 动作类型切换到“多媒体键”
2. 选择“播放/暂停”
3. 点击“确认配置”

### 冲突提示

如果你在创建新按键组时，里面有任意一个键已经属于别的按键组，系统会阻止保存，并提示：

`有按键和已有组合冲突，请确认后重新设置`

同时会补充：

- 冲突按键是哪个
- 它已经属于哪个按键组

### 进度条说明

顶部是三步进度：

1. 等待按键
2. 可以固定
3. 配置动作

含义如下：

- 没按键时：停在第 1 步
- 按下目标键后：进入第 2 步
- 固定并开始配置后：进入第 3 步

如果想重新开始当前流程，可以点击右上方的“清空状态”。

注意：

- 清空状态不会删除已经保存的按键组
- 它只会重置当前选择和进度

### 最小化到托盘

当你已经配置完成，希望安静地在后台运行时：

1. 点击窗口顶部的“最小化到托盘”
2. 主窗口会隐藏
3. 应用继续在菜单栏运行

此时你可以通过菜单栏图标：

- 显示主窗口
- 再次隐藏到托盘
- 退出 PadMapper

## 辅助功能权限

如果没有辅助功能权限，PadMapper 仍然可以：

- 识别按键
- 创建按键组
- 保存配置
- 显示命中与日志

但它不能真正向系统发送快捷键动作。

授权方法：

1. 打开应用
2. 展开“更多设置”
3. 点击“请求辅助功能权限”
4. 在系统设置中把 PadMapper 加入允许列表

如果未授权，运行反馈里通常只会记录日志，不会真的触发系统快捷键。

## 配置存储

当前配置默认保存在：

```text
~/Library/Application Support/PadMapper/profile.json
```

PadMapper 启动时会自动尝试加载这份配置。

## 开发说明

### 当前实现策略

当前版本优先保证原型可用，因此采用了比较直接的结构：

- `AppModel` 统一管理 UI 状态和应用主流程
- `InputDeviceService` 屏蔽真实 HID 输入与 mock 输入差异
- `ActionExecutor` 屏蔽快捷键 / 多媒体键执行细节

这意味着：

- 原型迭代速度快
- 功能链路清晰
- 但后续如果继续扩展多 Profile、动作链、按 App 切换，建议进一步拆分状态和运行时层

### 关于真实设备模式

当前真实设备接入基于 `IOHIDManager`，只监听选中的目标设备。

但请注意：

- 当前没有使用设备独占
- 当前不会完全阻断原始输入
- 这是一个现实可用的 MVP，但不是最终形态

## 已知问题

- 某些 HID 设备的物理键标识可能不够稳定
- 菜单栏托盘目前是基础版本，没有做开机启动
- 当前没有导入导出和多 Profile
- 当前没有设备独占，所以某些设备可能仍会把原始输入送给系统
